## Problem statement

v1.3.22 country expansion is functionally correct but
synchronously blocks the operator's HTTP request for up to
~30 seconds on fragmented countries (BR / IN). The Settings
UI's submit button stays disabled with a vague "Expanding (up
to ~10 min for large countries)..." label; the panel's
http.Server.WriteTimeout is bumped to 20 minutes specifically
to give this handler room. Both are sub-optimal:

- 20-min WriteTimeout exposes other endpoints to a longer
  hung-handler ceiling than necessary.
- Operator UX is "click button, watch spinner for 30s".
- A future panel reload mid-expansion (browser reload, panel
  restart) leaves the operator with no way to resume.

## Approach

**Async background job + polling.** The expand endpoint kicks
off a goroutine, returns 202 + job_id immediately. The
goroutine writes progress (chunks_done / chunks_total) to a
new `country_expansion_jobs` table. The frontend polls a job-
status endpoint until completion.

## Data model

New table, migration 030:

```sql
CREATE TABLE country_expansion_jobs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    country_code    TEXT NOT NULL,
    state           TEXT NOT NULL CHECK (state IN
                       ('queued', 'running', 'succeeded',
                        'failed', 'cancelled')),
    chunks_total    INTEGER NOT NULL DEFAULT 0,
    chunks_done     INTEGER NOT NULL DEFAULT 0,
    chunks_failed   INTEGER NOT NULL DEFAULT 0,
    cidr_committed  INTEGER NOT NULL DEFAULT 0,
    requested_count INTEGER NOT NULL DEFAULT 0,
    error_message   TEXT NOT NULL DEFAULT '',
    started_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    completed_at    TIMESTAMP,
    created_by      TEXT NOT NULL
);
CREATE INDEX idx_country_expansion_jobs_state ON country_expansion_jobs(state);
```

One job per expansion attempt. The existing
`country_ban_expansions` row is written on job success only;
a job is the transient progress shadow.

## Endpoints

```
POST  /api/security/countries/expand
      body: {country_code, duration, reason}
      -> 202 + { job_id, country_code, ... }
      (kicks off async job)

GET   /api/security/countries/jobs/{job_id}
      -> { job_id, state, chunks_done, chunks_total,
           cidr_committed, error_message, ... }

POST  /api/security/countries/jobs/{job_id}/cancel
      -> 200 + { cancelled: bool }
      (best-effort cancel; in-flight chunks may complete)
```

The DELETE / GET (list expansions) endpoints stay synchronous
-- they're cheap.

## Frontend

`CountryBansSection` switches to:

1. POST expand -> 202 with job_id.
2. Set submitting=true; start polling job status every 1s.
3. Render progress: "Expanding BR: 8 of 12 chunks (4500/5009 ranges)".
4. On state=succeeded: refresh table, toast success.
5. On state=failed: toast error with the job's error_message.
6. On panel reload mid-job: a "View jobs" surface in the
   Settings page picks up where the user left off.

Progress polling re-uses the existing fetch client; no
WebSocket / SSE infra needed.

## Tests

- Unit: job lifecycle (queued -> running -> succeeded /
  failed / cancelled), state transitions, idempotency.
- Unit: cancellation prevents future chunks from being
  pushed.
- Integration: scripts/smoke/country-block.sh continues to be
  the oracle. The smoke script handles the async flow:
  POST -> poll until done -> assert per-IP enforcement.
- New scripts/smoke/country-cancel.sh: starts a job,
  cancels mid-flight, asserts in-flight chunks did NOT
  commit.

## Open questions for the implementation session

- **Worker pool sizing.** One job at a time vs concurrent
  countries? Multiple concurrent jobs would compete for LAPI
  WAL locks (the v1.3.22 finding). Default to single-worker.
- **Job retention.** Drop jobs older than N days?
- **Restart recovery.** Panel restart mid-job: mark
  state='failed' with error_message='panel restarted' on
  next boot? Or attempt resume via the existing tracking
  row?
- **WriteTimeout rollback.** Once async lands, can we revert
  the panel http.Server.WriteTimeout back to 30s? Yes -- the
  expand endpoint returns 202 in <100ms.

## Why not stay synchronous

v1.3.22's chunked-sync path is correct, but it has three
operator-visible costs:

1. The 20-min WriteTimeout is a cross-cutting concern that
   raises the ceiling for every handler.
2. Operator can't multitask during a 30s expansion.
3. No restart recovery: a panel reload mid-expansion is
   indistinguishable from a panel crash, and the operator
   has no way to know if the half-finished state is safe to
   re-issue.

Async fixes all three for ~200 lines of new code (job table,
worker, status endpoint, polling-aware UI).

## When to schedule

Not urgent. v1.3.22's sync path works, the 20-min
WriteTimeout is a soft cap that doesn't degrade normal
operation. The async work makes most sense bundled with
v1.3.23+'s broader UX overhaul (banned-IPs / whitelist /
activity tabs from the v1.3.20-elevated-priority list),
where the same job-tracking infrastructure can serve other
long-running operations.
