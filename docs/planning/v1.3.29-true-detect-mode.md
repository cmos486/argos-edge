# v1.3.29 plan -- Per-host true_detect_mode (deferred from v1.3.27)

Status: planning. Gate-zero spike required before implementation.

Originally numbered v1.3.28; renamed to v1.3.29 when v1.3.28 was
claimed by the LAPI WAL fix. Same content; the deferred feature
now ships one cycle later.

## Background

`hosts.true_detect_mode` exists as a dormant column since
migration 028 (v1.3.19). The model.go field is annotated with a
profiles.yaml-based intended implementation; v1.3.27 originally
planned a Caddy per-host `appsec_url` emit instead.

v1.3.27 pre-flight verification (this is the sixth case in the
upstream-behaviour pattern memo) confirmed the Caddy path is
architecturally impossible against the pinned plugin:

```go
// hslatman/caddy-crowdsec-bouncer @ appsec/appsec.go (v0.12.1, latest)
type Handler struct {
    logger   *zap.Logger
    crowdsec *crowdsec.CrowdSec
}
func (h *Handler) UnmarshalCaddyfile(d *caddyfile.Dispenser) error {
    return nil  // no Caddyfile config accepted
}
func (h *Handler) Provision(ctx caddy.Context) error {
    crowdsecAppIface, err := ctx.App("crowdsec")  // global
    ...
}
```

The global `apps.crowdsec.appsec_url` is `string`, not a map. No
per-handler override field exists; the per-route handler is a
no-config wrapper that always reads the singleton URL.

## Two paths to evaluate at v1.3.29 kickoff

### Path C (re-spike): profiles.yaml whitelist

The model.go comment claims this works. v1.3.27 user judgement
rejected it as upstream-unsupported, but the rejection was not
empirically verified.

CrowdSec profiles.yaml supports `filter` (CEL on Alert objects)
and `decisions` (action). A profile with `filter:
Alert.GetSourceIp() == "X.X.X.X"` and empty `decisions` would
suppress decision-creation for that source.

**Gate-zero spike (~30 min)**:

1. Pick two test hosts (e.g., `detect.example.com`,
   `block.example.com`).
2. Add a profiles.yaml entry filtering AppSec alerts whose
   source/host matches `detect.example.com`.
3. Fire a malicious request (suspicious User-Agent, SQLi
   payload) at each host from the same client IP.
4. `cscli alerts list`: should show two alerts (one per host).
5. `cscli decisions list --ip <client>`: should show only one
   decision (the block-host alert fed it; the detect-host alert
   was suppressed by the profile filter).
6. If gate-zero PASSes, this is the v1.3.29 path: ship a
   profiles.yaml writer + reconciler trigger that rebuilds the
   profile from `hosts.true_detect_mode = true` rows.

**Open question for path C**: AppSec alert fields. Whether
`Alert.GetSourceIp()` resolves to client IP, host header, or
upstream IP determines whether the filter can reliably target a
specific host. Empirical only -- no docs.

### Path PR: upstream contribution to caddy-crowdsec-bouncer

Add an optional `appsec_url` field to `appsec.Handler` that
overrides the global. Mechanical change:

```go
type Handler struct {
    AppSecURL string `json:"appsec_url,omitempty"`
    logger    *zap.Logger
    crowdsec  *crowdsec.CrowdSec
}

func (h *Handler) Provision(ctx caddy.Context) error {
    crowdsecAppIface, err := ctx.App("crowdsec")
    h.crowdsec = crowdsecAppIface.(*crowdsec.CrowdSec)
    if h.AppSecURL == "" {
        // fall back to global
        h.AppSecURL = h.crowdsec.AppSecURL
    }
    h.logger = ctx.Logger(h)
    return nil
}
// ServeHTTP uses h.AppSecURL instead of h.crowdsec.AppSecURL
```

**Open questions for path PR**: maintainer responsiveness
unknown; v0.12.1 was released 2026-04-13. If the PR stalls, the
project would need to fork + replace. Risk of being stuck on a
fork indefinitely.

## Decision criteria for path selection (run at v1.3.29 start)

| Criterion | Path C wins | Path PR wins |
|---|---|---|
| Gate-zero spike PASSes | Yes -> ship | n/a |
| Gate-zero spike FAILs (filter can't disambiguate hosts) | n/a | Yes |
| Maintainer activity on PRs in last 30d | n/a | Recent merges = path PR viable |
| Operator wants a no-fork solution | Yes | No |

Default if both feasible: path C, because zero upstream
dependency.

## Smoke gate v1.3.29 (independent of path)

`scripts/smoke/true-detect-mode.sh`:

1. Pick two test hosts (one detect, one block).
2. Fire a malicious request at each.
3. `cscli alerts list`: two alerts visible.
4. `cscli decisions list --ip <client>`: one decision (block
   host fed it), zero from the detect host.
5. Toggle off detect-mode via API.
6. Re-fire malicious request at the formerly-detect host.
7. `cscli decisions list --ip <client>`: a second decision
   appears (the previously-detect host now feeds bans).
8. Cleanup: re-enable both hosts to default.

## Out-of-scope for v1.3.29

- Re-evaluating the Caddy path. Plugin v0.12.x line is
  architecturally fixed. A v0.13+ release with per-handler
  config would be the only Caddy-side path; not present today.
