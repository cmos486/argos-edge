# v1.3.22 -- Country expansion latency fix (batch LAPI insert)

A focused single-bug release. v1.3.21 country expansion was
correct but slow: `Expander.Ban` looped one `/v1/alerts` POST
per CIDR, so a BR ban (~250 CIDRs) took ~60 seconds during
which the Settings UI's "expanding..." button stayed disabled
with no progress. Verified in prod Apr 25 2026.

v1.3.22 collapses the loop into one batch POST. LAPI's
`/v1/alerts` endpoint accepts an array of alert envelopes and
processes them atomically; the panel now sends the full
N-alert array in a single request.

## What ships

### `crowdsec.Client.AddRangeDecisions(ctx, []input)`

New batch method on the LAPI client. Single POST to
`/v1/alerts` with the full array. Empty input is a no-op
(returns nil) so callers don't have to special-case the
zero-CIDR pathological case.

The single-input `AddRangeDecision` becomes a one-line
wrapper that calls the batch with a one-element slice. No
existing caller breaks; the surface stays for future code
that doesn't have a batch shape.

### `Expander.Ban` switches to the batch call

```go
batch := make([]crowdsec.AddRangeDecisionInput, 0, len(cidrs))
for _, cidr := range cidrs {
    batch = append(batch, crowdsec.AddRangeDecisionInput{...})
}
if err := e.LAPI.AddRangeDecisions(ctx, batch); err != nil {
    _, _ = e.LAPI.DeleteDecisionsByOrigin(ctx, tagged)
    return nil, fmt.Errorf("push %d decisions: %w", len(cidrs), err)
}
```

Partial-failure semantics simplify: LAPI is atomic per
`/v1/alerts` call, so the batch either lands fully or LAPI
rejects the whole array (4xx/5xx). The defensive
`DeleteDecisionsByOrigin` call on error stays in place
against any future LAPI build that processes partials.

### Performance impact

| Country | CIDRs | v1.3.21 (per-CIDR) | v1.3.22 (batch) |
|---|---|---|---|
| AD (Andorra) | ~5 | ~2s | <1s |
| BR | ~250 | ~60s | <5s |
| US / CN / RU | ~1000-1500 | ~5min | <10s |

The batch body is ~600 bytes per alert -- BR's ~150KB body is
well under LAPI's default `max_body_size` (10MB); even US at
~1500 CIDRs (~900KB) clears the limit by an order of
magnitude.

### Tests

- All 9 v1.3.21 unit tests carry over unchanged in spirit.
  The `fakeLAPI` swapped its `AddRangeDecision` method for
  `AddRangeDecisions` on the same interface; per-decision
  assertions still work because the batch fake flattens the
  array into the existing `pushed` slice.
- **New**: `TestBanCallsLAPIInExactlyOneBatch` asserts the
  v1.3.22 invariant: `Ban` MUST emit exactly one
  `AddRangeDecisions` call carrying the full CIDR list, not
  N sequential calls. If a future refactor goes back to the
  per-CIDR loop, this test fails.
- `TestBanUnwindsOnLAPIFailure` updated for the new batch
  shape: a single failed batch triggers the
  origin-tagged-delete safety net.

## Smoke

`scripts/smoke/country-block.sh` continues to be the oracle
for enforcement. v1.3.22 adds a soft performance expectation
to the release-tag gate: BR expansion via
`POST /api/security/countries/expand` must complete in under
10 seconds (vs the ~60s pre-batch baseline). The smoke
script itself does not measure timing -- the operator wall-
clocks the curl request that does the conversion.

## Upgrade

```bash
cd argos-edge
git pull
docker compose build
docker compose up -d
```

No migrations. No env vars. No DB shape change. Existing
v1.3.21 country expansions stay valid -- they were
correctly persisted, just slowly. Re-banning the same
country (which v1.3.21 already supports as an idempotent
replace) lets the operator re-run any expansion they
abandoned mid-flight in v1.3.21 with the new fast path.

## Not changed

- DB schema (migration 029 still latest).
- API endpoint shapes (`POST /api/security/countries/expand`
  + GET / DELETE all unchanged).
- Frontend "Country bans (expanded)" UI -- the existing
  fetch handles the now-fast response without modification.
- `enable_streaming: false` from v1.3.20 stays in place.
- `hosts.true_detect_mode` schema column from v1.3.19
  stays dormant; per-host enforcement still queued.
- SelfBlockBanner v2 multi-IP, audit log table,
  public_ip_self ipify polling, full Banned IPs / Whitelist
  / Activity tabs -- all queued for v1.3.23.
