    v1.3.21 plan -- Country bans via panel-side Range expansion¶

    Status: PLAN ONLY. Implementation deferred to a dedicated session. This document captures the design so the next session has a self-contained brief.

    Problem statement¶

    Country-based geo-blocking has been documented as working since v1.3.17 but has never functioned in any v1.3.x release. The upstream hslatman/caddy-crowdsec-bouncer plugin does not handle scope=Country decisions in either stream OR live mode (verified Apr 25 2026 against plugin commit f1e77b2 -- see docs/release-notes/v1.3.20.md for the full upstream-source citation, plus the matching memory entry).

    v1.3.20 attempted a one-flag fix (enable_streaming: false) based on a wrong assumption about live mode's behavior. That release shipped to main but is not tagged; the smoke script scripts/smoke/country-block.sh correctly reports FAIL.

    Approach¶

    Panel-side expansion. When the operator bans Country=BR, the panel:

    1. Looks up the BR CIDR list in an embedded GeoLite2-Country MMDB.
    2. Pushes each CIDR to LAPI as a --scope Range decision.
    3. Persists a tracking row mapping country code -> the list of decision IDs that were created.

    The plugin handles Range decisions natively in both stream and live mode, so country blocking actually works.

    Revocation reads the tracking row, deletes every Range decision it created, and removes the tracking row. The MMDB version at creation time is recorded so a country-list update on a future MMDB refresh can be reconciled (drop removed CIDRs, add new ones, update the tracking row).

    Data model¶

    New table, migration 029:

    CREATE TABLE country_ban_expansions (
    id                          INTEGER PRIMARY KEY AUTOINCREMENT,
    country_code                TEXT NOT NULL,        -- ISO 3166-1 alpha-2
    decision_ids                TEXT NOT NULL,        -- JSON array of LAPI decision IDs
    cidr_count                  INTEGER NOT NULL,     -- denormalised for the UI list
    reason                      TEXT NOT NULL DEFAULT '',
    duration                    TEXT NOT NULL,        -- Go duration string ("4h", "24h", ...)
    created_at                  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_by                  TEXT NOT NULL,        -- panel user that issued the ban
    mmdb_version_at_creation    TEXT NOT NULL,        -- MMDB build_epoch or string version
    UNIQUE(country_code)                              -- one active expansion per country
);

    Down migration drops the table. No backfill needed -- this is a new feature.

    The UNIQUE(country_code) constraint is deliberate: an operator-issued country ban is an "intent", not a fact about LAPI. If they ban BR twice, that's a no-op idempotent intent, not two stacked expansions. Re-issuing replaces the existing expansion (delete old decisions, insert new ones) so the MMDB version stays current.

    Service: CountryBanExpander¶

    New package backend/internal/security/country/. Public API:

    // Expand a country code into LAPI Range decisions and
// persist the tracking row. Idempotent: a second call with
// the same country_code refreshes the expansion.
func (e *CountryBanExpander) Ban(
    ctx context.Context,
    countryCode string,    // "BR", "RU", ...
    duration time.Duration,
    reason string,
    actor string,          // panel user
) (*CountryBanResult, error)

// Revoke an existing expansion: delete every LAPI decision
// it created, then drop the tracking row.
func (e *CountryBanExpander) Revoke(
    ctx context.Context,
    countryCode string,
) error

// Reconcile an existing expansion against the current MMDB:
// drop CIDRs that no longer map to this country, add new
// ones, update the tracking row's mmdb_version. Called on
// MMDB refresh.
func (e *CountryBanExpander) Reconcile(
    ctx context.Context,
    countryCode string,
) (*CountryBanReconcileResult, error)

// List active expansions for the UI Banned IPs / Threats
// tab.
func (e *CountryBanExpander) List(
    ctx context.Context,
) ([]CountryBanExpansion, error)

    Internally:

    • MMDB lookup via github.com/oschwald/geoip2-golang (already a dep -- see backend/internal/geoip/).
    • LAPI decision push via the existing crowdsec.Client methods (AddDecision, DeleteDecision); no new LAPI surface.
    • Errors wrapped with operation context (country: ban BR: lapi push: ...) so failures are diagnosable from panel logs.

    MMDB lifecycle¶

    argos already ships a GeoLite2-Country MMDB for the geoip enrichment feature (backend/internal/geoip/). The country- expansion service reuses it. Refresh is the existing monthly cron at 0 3 5 * * -- no new lifecycle, no new env vars.

    The MMDB version recorded at expansion time is the file's build_epoch field, which is what oschwald/geoip2-golang already exposes. On the monthly cron post-refresh, a follow-up reconcile pass walks every active expansion and calls Reconcile(...) to keep CIDRs current.

    License-key impact: the panel already documents the MaxMind license key requirement for refresh. v1.3.21 inherits that contract. Operators without a license key keep the bundled snapshot and accept that country expansions stale slowly.

    Lifecycle endpoints¶

    Three new endpoints under the existing /api/security/ prefix (continues the v1.3.19 namespace):

    POST /api/security/countries/expand
    body: { country_code, duration, reason }
    -> 201 + { country_code, cidr_count, decision_ids, ... }

GET  /api/security/countries
    -> [ { country_code, cidr_count, created_at, ... }, ... ]

DELETE /api/security/countries/:cc
    -> 204

    Audit-logged: each call writes an entry to the audit log table (which itself is queued for v1.3.21 -- see project_v1320_scope.md).

    Tests¶

    Unit¶

    • country_ban_expansions table CRUD, including the UNIQUE(country_code) idempotency.
    • MMDB lookup returns expected CIDRs for a known country (use a fixture MMDB committed to the repo, NOT the live GeoLite2 DB -- the test must be deterministic and not depend on which MMDB version the dev machine has cached). Pick a small country with a stable CIDR list for the fixture; document the version baked into the fixture.
    • Reconcile correctly diffs old vs new CIDR lists across MMDB versions: drops removed, adds new, leaves unchanged.
    • Revoke deletes the LAPI decisions in the tracking row even when LAPI returns 404 for some IDs (idempotent cleanup; do not bail on stale IDs).

    Integration¶

    • scripts/smoke/country-block.sh PASSes after v1.3.21 deploy. This is the oracle of success for the entire feature. The script is unchanged from v1.3.20 -- it has always been correct; only the implementation under test has been wrong.
    • A dedicated scripts/smoke/country-revoke.sh (new) that bans a country, verifies 403, revokes, verifies 200. Exercises the full lifecycle end-to-end.

    What v1.3.21 explicitly does NOT add¶

    • Per-host country exemptions. Out of scope; defer.
    • Country whitelist (only allow X, Y). Same upstream limitation -- the plugin doesn't understand ALLOWED-Country either. Defer or punt.
    • UI for editing the expansion CIDR list manually. Operators who want fine-grained control use the Range bans path directly via cscli.

    Smoke gate¶

    Smoke must PASS in the live prod stack before tagging v1.3.21. Working agreement clarified after v1.3.20 (see the matching memory entry). Unit tests are necessary but not sufficient for upstream-behavior fixes.

    Why not "switch bouncer plugin"¶

    The agent investigation considered switching to a different Caddy CrowdSec bouncer plugin that handles Country natively. None of the alternatives have the maturity of hslatman/caddy-crowdsec-bouncer; the most mature alternative is significantly less battle-tested. Panel-side expansion keeps argos on the well-trodden plugin and adds value the plugin will never have (country-list lifecycle tracking with MMDB-version reconciliation).

    Why not "send the upstream PR"¶

    Worth doing in parallel as good citizenship, but argos cannot block on it -- upstream PRs to that repo have a multi-month merge cadence and the surface (radix-tree lookup keyed on IP) is fundamentally not built for non-IP scopes. A clean PR would be a major redesign of the plugin's decision store, which is beyond the scope of a feature argos owners need now.

    A v1.3.21+N follow-up should consider opening an issue upstream documenting the gap, even if argos's own fix ships without it.
